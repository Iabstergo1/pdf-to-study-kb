# 阶段 2B-3：个人高相关型小节薄切片试跑计划

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** 验证个人高相关型小节的处理闭环：从 PDF 原文提取 → 生成讲义草稿（强化个人知识体系连接） → 校验通过 → 审校通过。重点验证与 2B-1（概念型）、2B-2（公式风险型）的差异处理。

**Architecture:** 5 个任务顺序执行：原文提取 → 草稿生成 → 结构校验 → 内容审校 → manifest 回填。在讲义生成阶段强化"与个人知识体系的连接候选"章节。如遇 revise/reject 则停下报告，不自动重试。

**Tech Stack:** pymupdf (PDF 提取), Python (validator), Markdown (讲义/报告)

## 前置决策

### Python 环境

**显式使用完整路径：**
```
python
```

### Windows 工具选择

**问题：** Claude Code 的 Bash 工具底层是 Git Bash (MSYS2)，在处理含中文的 Windows 路径时会崩溃（`fatal error - add_item`，Exit code 5）。

**规则：**
1. **优先使用原生工具**：Glob（文件搜索）、Grep（内容搜索）、Read（读取）、Edit（编辑）——这些不经过 Bash，无路径问题。
2. **需要执行命令时**：直接调用 `pwsh`（PowerShell 7），不要通过 Git Bash 调用 PowerShell。
3. **禁止**：不要用 Bash 工具执行 `powershell -Command "..."` 或 `Select-String` 等 PowerShell 命令。

**示例：**
```powershell
# 错误：通过 Git Bash 调用 PowerShell（会因中文路径崩溃）
# bash: powershell -Command "Select-String -Path 'books/博弈论白皮书/...'"

# 正确：直接用 pwsh
pwsh -Command "Select-String -Path 'books/博弈论白皮书/...'"

# 更好：用原生 Grep 工具代替 Select-String
# Grep pattern="GTW-006-05" path="books/博弈论白皮书/config/section-manifest.yaml"
```

### Git

**Stage 2B-3 不要求 Git。** commit 步骤标记为可选。

### Manifest Order 与原文可见章节号

**已知语义差异：** section-manifest.yaml 中 GTW-006-05 的 `source_order` 为 `3.1.5`，但 PDF 原文中该小节显示为 `6.5`。这是框架设计中 `book_order`（按书结构编排）与 PDF 可见章节号的差异。

**本阶段决策：**
- 在 source-slice.md 元数据头中同时记录 manifest order 和原文可见章节号
- 在讲义 frontmatter 中 `book_order` 使用 manifest 值（`3.1.5`）
- 不批量修改 manifest，仅在本节记录差异

### Task 4: 审校讲义草稿

**Files:**
- Create: `books/博弈论白皮书/pipeline-workspace/reviews/GTW-006-05/review-report.md`
- Create: `books/博弈论白皮书/pipeline-workspace/reviews/GTW-006-05/review-decision.yaml`

**依赖：** Task 3 通过

- [ ] **Step 1: 创建 reviews 目录**

```powershell
pwsh -Command "New-Item -ItemType Directory -Force -Path 'books/博弈论白皮书/pipeline-workspace/reviews/GTW-006-05'"
```

- [ ] **Step 2: 按 review skill 进行审校**

遵循 `.claude/skills/section-lesson-review/SKILL.md` 的规则，但针对个人高相关型做以下强化：

**额外审校项（个人连接专项）：**

1. **连接候选质量**
   - 连接候选是否具体（非泛泛而谈）
   - 桥接对象是否明确（指向个人知识体系中的具体概念/工具）
   - 桥接理由是否充分（为什么值得连接）
   - 桥接方式是否可操作（读者能据此行动）

2. **连接候选覆盖度**
   - 是否覆盖至少 3 个维度
   - 是否有工具层面的连接
   - 是否有流程或方法论层面的连接

3. **实践指导性**
   - 原文中的工具推荐是否准确传达
   - 工作流程描述是否清晰
   - 读者能否据此实际操作

**决策规则（同 2B-1）：**

1. **reject**：忠实性 FAIL 或结构完整性 FAIL
2. **revise**：任意维度 FAIL 或 WARN > 3
3. **accept**：无 reject/revise 条件

- [ ] **Step 3: 生成 review-decision.yaml**

内容格式：
```yaml
section_id: GTW-006-05
reviewer: claude
review_date: <YYYY-MM-DD>
decision: <accept|revise|reject>
scores:
  faithfulness: <PASS|WARN|FAIL>
  learnability: <PASS|WARN|FAIL>
  importance: <PASS|WARN|FAIL>
  source_return: <PASS|WARN|FAIL>
  structure: <PASS|FAIL>
required_fixes: []
warnings: []
notes: ""
```

- [ ] **Step 4: 生成 review-report.md**

使用 `templates/review-report.template.md` 格式，填写实际审校结果。

- [ ] **Step 5: 根据决策结果处理**

**如果 decision = accept：**
- 记录成功
- 进入 Task 5（GTW-006-05 manifest 回填）
- 不发布到 study-kb（本阶段禁止）

**如果 decision = revise：**
- 记录 required_fixes
- 停下并报告，等待指示
- 不自动重试
- 不更新 manifest

**如果 decision = reject：**
- 记录拒绝理由
- 停下并报告，等待指示
- 不自动重试
- 不更新 manifest

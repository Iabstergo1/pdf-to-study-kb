# 阶段 2B-2：公式风险小节薄切片试跑计划

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** 验证公式/推导型小节的处理闭环：从 PDF 原文提取 → 公式风险标记 → 生成讲义草稿 → 校验通过 → 审校通过。重点验证与 2B-1（概念型）的差异处理。

**Architecture:** 7 个任务顺序执行：slicer 泛化 → 回填 2B-1 manifest → 原文提取 → 草稿生成 → 结构校验 + 人工检查 → 内容审校 → manifest 回填。在原文提取和讲义生成阶段增加公式风险处理逻辑。如遇 revise/reject 则停下报告，不自动重试。

### Windows 工具选择

**问题：** Claude Code 的 Bash 工具底层是 Git Bash (MSYS2)，在处理含中文的 Windows 路径时会崩溃（`fatal error - add_item`，Exit code 5）。

**规则：**
1. **优先使用原生工具**：Glob（文件搜索）、Grep（内容搜索）、Read（读取）、Edit（编辑）——这些不经过 Bash，无路径问题。
2. **需要执行命令时**：直接调用 `pwsh`（PowerShell 7），不要通过 Git Bash 调用 PowerShell。
3. **禁止**：不要用 Bash 工具执行 `powershell -Command "..."` 或 `Select-String` 等 PowerShell 命令。

### Git

**Stage 2B-2 不要求 Git。** commit 步骤标记为可选。

### Manifest 状态契约

**问题：** Stage 2B-1 完成后 GTW-002-01 的 review-decision.yaml 为 `decision: accept`，但 section-manifest.yaml 中该条目仍为 `status: registered`。这是 2B-1 的遗留状态分裂。

**状态记录的职责分工：**

- **section-manifest.yaml** 是阶段恢复、编排和状态登记的主入口。所有 section 的当前状态以此文件为准。
- **review-decision.yaml** 是审校证据，也是 manifest 状态更新的依据。当审校通过后，orchestrator 据此更新 manifest。
- 两者不一致时视为**待回填/待对账**，不应长期接受状态分裂。发现不一致时应立即修正 manifest。

**本阶段决策：**

1. **GTW-002-01 回填（前置任务）：** 在 GTW-005-01 相关任务开始前，单独将 section-manifest.yaml 中 GTW-002-01 的 `status` 从 `registered` 更新为 `reviewed`。依据：2B-1 的 review-decision.yaml 已记录 `decision: accept`。此为独立 Task 1，不与 GTW-005-01 的回填合并。

2. **GTW-005-01 状态回填：** 本阶段 Task 5 审校通过（decision = accept）后，更新 section-manifest.yaml 中 GTW-005-01 的 `status` 为 `reviewed`。此为独立 Task 6。

3. **状态链对齐：** 本阶段更新的状态值为 `reviewed`（而非 `accepted`），与框架定义的 `registered → extracted → drafted → reviewed → accepted → published` 状态链一致。`accepted` 状态的转换应由后续阶段（如发布阶段）负责。

**与"manifest 驱动"原则的一致性：** 本阶段的 manifest 更新由 orchestrator 角色（本计划的执行者）在审校门禁通过后执行，符合框架中"orchestrator 推进阶段、更新状态"的设计。Worker（section-worker、section-reviewer）仍只写 staging 和 reviews 目录。

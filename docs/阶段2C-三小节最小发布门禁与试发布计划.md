## 3. 发布门禁检查（Pre-publish Gate）

### 3.1 检查项

| 检查项 | 检查方式 | 阻断级别 |
|--------|----------|----------|
| review-decision.yaml 存在 | 文件存在性 | block |
| review-decision.yaml 中 decision=accept | 内容校验 | block |
| section-manifest.yaml 中 status=reviewed | 内容校验 | block |
| section-lesson-draft.md 存在 | 文件存在性 | block |
| source-slice.md 存在 | 文件存在性 | warn |

### 3.2 决策：发布后 status 字段如何更新

**选项 A（推荐）**：发布后同时更新两个字段
- `publish_status: published`
- `status: published`

理由：status 表示该小节在流水线中的最终状态，发布完成即表示该小节生命周期结束。简化后续状态机。

**选项 B**：只更新 publish_status
- `publish_status: published`
- `status: reviewed`（保持不变）

理由：status 保留为 reviewed，表示"内容已审阅"这一事实，publish_status 单独追踪发布状态。两字段解耦。

**决策结果**：采用 **选项 A**，发布后 status 和 publish_status 均更新为 published。理由：试点阶段保持状态机简洁，避免两个字段语义重叠导致混淆。

## 4. 发布流程（Manual Script Steps）

### Step 1：运行门禁检查

对每个待发布小节执行：
1. 检查 `reviews/{section-id}/review-decision.yaml` 存在且 `decision: accept`
2. 检查 `section-manifest.yaml` 中对应条目 `status: reviewed`
3. 检查 `staging/{section-id}/section-lesson-draft.md` 存在

任一检查失败则跳过该小节，记录到发布报告。

### Step 2：发布转换（非直接复制）

从 staging 草稿生成发布版本，输出到 `study-kb/Section-Lessons/{section-id}.md`。

**转换规则**：
1. 保留正文内容（从第一个 `# ` 标题开始到文件末尾）
2. 保留或重建完整 frontmatter，必须包含以下字段：
   - `id`, `type`, `source_title`, `source_locator`, `book_order`
   - `importance`, `difficulty`, `formula_risk`
   - `review_status`, `generation_stage`
   - 若原草稿有 `personal_kb_relevance`，也要保留
3. 更新状态字段值：
   - `review_status: reviewed`（草稿中为 `draft`）
   - `generation_stage: published`（草稿中为 `draft`）
4. 不要把 staging 草稿的 draft 状态带入 study-kb

**最终期望值表**（发布后状态）：

| 位置 | 字段 | 期望值 |
|------|------|--------|
| 讲义 frontmatter | `review_status` | `reviewed` |
| 讲义 frontmatter | `generation_stage` | `published` |
| manifest | `status` | `published` |
| manifest | `publish_status` | `published` |

> **重要**：不要把讲义的 `review_status` 与 manifest 的 `status` 做值相等校验。两套字段虽然语义相关，但命名和取值范围不同，发布时只需确保各自达到期望值即可。

**文件名规则**：直接使用 section-id 作为文件名（如 `GTW-002-01.md`），保持简洁可预测。

### Step 3：更新 manifest 状态

将 `section-manifest.yaml` 中对应条目更新为：
```yaml
status: published
publish_status: published
```

- 文件非空
- 包含一级标题（`# `）
- frontmatter 中包含 `source_locator:` 字段
- 正文中包含 `## 原文定位` 章节

### 5.3 Manifest 一致性检查

验证 `section-manifest.yaml` 中：
- 3 个小节的 `status: published`
- 3 个小节的 `publish_status: published`

## 8. 执行顺序

```
1. 运行门禁检查（3 个小节）
   ↓ 通过
2. 发布转换：staging 草稿 → study-kb/Section-Lessons/{section-id}.md
   ↓
3. 更新 manifest 状态
   ↓
4. 生成试点阅读地图
   ↓
5. 生成覆盖报告（从 manifest 实时统计）
   ↓
6. 生成风险清单
   ↓
7. 运行 validator（双链、完整性、一致性）
   ↓ 通过
8. 生成发布报告
   ↓
9. 完成
```

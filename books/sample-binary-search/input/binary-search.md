# 二分查找（Binary Search）

> 这是随仓库分发的**样例源**，只为克隆后快速跑通 ingest 流程。可随时删除或替换成你自己的文档。

## 问题

在一个**已排序**的数组里查找目标值的下标。逐个扫描是 O(n)，有没有更快的办法？

## 核心思想

每次比较中间元素，把搜索区间**对半砍**：目标比中间小就只看左半、比中间大就只看右半、相等即命中。每一步把候选数量减半，所以最多比较 ⌈log₂ n⌉ 次。

## 复杂度

- 时间：O(log n)——每步区间减半。
- 空间：迭代实现 O(1)；递归实现 O(log n)（调用栈）。
- **前提**：数组必须有序，否则"砍掉半边"的推理不成立。

## worked example

在 `[2, 5, 8, 12, 16, 23, 38]`（7 个元素）里找 `23`：

1. lo=0, hi=6，mid=3 → a[3]=12 < 23，去右半，lo=4。
2. lo=4, hi=6，mid=5 → a[5]=23，命中，返回下标 5。

两步搞定；逐个扫描要 6 步。

## 常见坑

- **中点溢出**：用 `mid = lo + (hi - lo) // 2` 而不是 `(lo + hi) // 2`，避免大数相加溢出。
- **区间约定**：闭区间 `[lo, hi]` 与半开区间 `[lo, hi)` 的循环条件和边界更新不同，混用是越界 / 死循环的常见来源。
